| 영역 | 기술 |

## 아키텍처

### 백엔드 — Layered Architecture (헥사고날 유사)

```
api (Controller) → application (Service) → domain (Entity) → infrastructure (Repository)
```

- 도메인별 패키지 분리: `auth`, `post`, `comment`, `file`
- Snowflake ID 기반 분산 ID 생성

### 프론트엔드 — Feature-Sliced Design

```
features/ → shared/ → widgets/
```

- TanStack Query (서버 상태) + Zustand (클라이언트 상태)


## 주요 기능

- **인증**: JWT 액세스(15분) + HttpOnly 리프레시(7일) + 토큰 로테이션
- **게시글**: CRUD + 커서 기반 페이지네이션 + 무한 스크롤
- **좋아요**: 유니크 제약 + 동시성 충돌 처리
- **댓글**: 최대 2depth 트리 구조 (parent_id 기반)
- **파일**: 멀티파트 업로드 + 소유권 검증 (5MB 제한)
- **인기글**: Kafka → Redis 집계

## API 엔드포인트

### 게시글
- `GET /api/v1/posts` — 목록 (커서 페이지네이션)
- `POST /api/v1/posts` — 작성 (인증 필요)
- `GET /api/v1/posts/{postId}` — 상세 조회
- `PUT /api/v1/posts/{postId}` — 수정 (인증 필요)
- `DELETE /api/v1/posts/{postId}` — 삭제 (인증 필요)
- `GET /api/v1/posts/popular` — 인기글 목록

### 좋아요
- `POST /api/v1/posts/{postId}/likes` — 추가
- `DELETE /api/v1/posts/{postId}/likes` — 제거

### 댓글
- `GET /api/v1/posts/{postId}/comments` — 목록
- `POST /api/v1/posts/{postId}/comments` — 작성
- `PUT /api/v1/posts/{postId}/comments/{commentId}` — 수정
- `DELETE /api/v1/posts/{postId}/comments/{commentId}` — 삭제

### 파일
- `POST /api/v1/files/upload` — 업로드
- `GET /api/v1/files/{fileId}` — 다운로드

### 인증
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`

## 잘 구현된 부분

1. **N+1 방지** — 배치 집계 쿼리로 다수 게시글 조회 시 성능 보장
2. **키셋 페이지네이션** — OFFSET 방식 대비 O(1) 성능
3. **보안** — BCrypt 라운드 12, JWT 검증, 파일 소유권 체크, HttpOnly 쿠키
4. **테스트** — 통합 테스트 + 단위 테스트 + E2E 전 계층 커버
5. **401 자동 재시도** — Request Collapsing으로 동시 refresh 중복 방지
6. **선택적 인프라** — `APP_REDIS_ENABLED=false`, `APP_KAFKA_ENABLED=false`로 로컬 실행 가능
7. **토큰 로테이션** — 리프레시 토큰 갱신 시 기존 폐기로 토큰 탈취 피해 최소화

## 개선 권장 사항

### High Priority (즉시 적용)

1. **프로덕션 JWT 시크릿 교체 필수**
   - 현재 `APP_JWT_SECRET=change-me-in-production...` 기본값 위험
   - 배포 전 반드시 강력한 랜덤 시크릿으로 교체

2. **Swagger/OpenAPI 문서 부재**
   - `springdoc-openapi-starter-webmvc-ui` 의존성 추가 권장
   - 프론트엔드 팀 협업 및 외부 연동 시 필수

3. **예외 클래스 세분화**
   - 현재 `ApiException` 단일 클래스 → `EntityNotFoundException`, `ForbiddenException` 등 분리
   - 클라이언트 에러 처리 정밀화 가능

### Medium Priority (단기)

4. **낙관적 잠금 미적용**
   - 동시 편집 시 Last-Write-Wins 문제 발생
   - `Post` 엔티티에 `@Version` 필드 추가 권장

5. **Redis 캐시 TTL/무효화 전략 미흡**
   - 명시적 TTL 설정 및 이벤트 기반 캐시 무효화 필요

6. **구조화 로깅 미적용**
   - Logback + Logstash JSON 인코더 적용 권장
   - 로그 수집/분석 시스템 연동 용이

### Low Priority (장기)

7. **파일 저장소 확장**
   - 현재 로컬 디스크만 지원
   - S3 / GCS 호환 객체 스토리지 지원 추가

8. **모니터링 부재**
   - Micrometer + Prometheus + Grafana 스택 도입 권장
   - Spring Actuator는 최소 설정(health, info)만 활성화 중

9. **CQRS 패턴**
   - 대규모 트래픽 시 읽기 모델 분리로 조회 성능 향상

## 총평

**대규모 애플리케이션의 모범 사례를 잘 따르고 있는 프로젝트**입니다.

- Spring Boot + React 풀스택 구조가 명확히 분리됨
- 성능 최적화(커서 페이지네이션, 배치 쿼리, Redis 캐싱)가 탄탄함
- 보안(JWT, HttpOnly 쿠키, 권한 검증)이 잘 구현됨
- 비동기 처리(Kafka 이벤트) 설계가 확장성 있음
- 전 계층 테스트 커버리지 보유

즉시 적용 가능한 개선 사항(시크릿 보안, API 문서화)만 반영해도 프로덕션 수준으로 충분히 활용 가능합니다.
